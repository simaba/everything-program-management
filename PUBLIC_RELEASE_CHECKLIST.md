# Public Release Checklist

Use this checklist before changing the repository visibility to public.

## Required before publishing

- [ ] Confirm all examples are fictional, generic, or fully sanitized.
- [ ] Confirm no real employer, customer, vendor, supplier, or colleague information is committed.
- [ ] Confirm no confidential project names, roadmaps, escalations, delivery risks, audit findings, or steering materials are included.
- [ ] Confirm no real RAID logs, decision memos, stakeholder maps, HR material, procurement details, financial details, or legal details are included.
- [ ] Confirm no non-public product details, architecture, defects, launch gates, or internal approval paths are included.
- [ ] Confirm no API keys, tokens, credentials, connector details, account identifiers, private paths, or emails are present.
- [ ] Confirm examples teach structure without exposing real work.
- [ ] Confirm the license, README, and quick-start commands are accurate.

## Recommended before promotion

- [ ] Add richer fictional worked examples for stakeholder maps and decision memos.
- [ ] Add a public-safety note to any example folder that may attract real artifacts.
- [ ] Add CSV import/export helpers only after sample data is sanitized.
- [ ] Run tests locally after a fresh clone.

## Final manual review

Before publishing, search the repo for company names, project names, internal acronyms, colleague names, customer names, vendor names, emails, tokens, keys, private paths, and real PM artifacts.

This checklist is a publication aid, not a security guarantee.
