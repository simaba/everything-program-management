# AGENTS.md

Compatibility entry point for agent harnesses that read `AGENTS.md`. The working guidance mirrors `CLAUDE.md`.

## Purpose

Use this repository to produce decision-ready program-management artifacts and structured data. Avoid generic management prose, performative framework naming, and templates filled with invented certainty.

## Operating principles

1. Lead with the decision, changed condition, recommendation, or outcome.
2. Separate measured facts, estimates, assumptions, interpretations, and unknowns.
3. Use named methods only when they add real constraints; do not name-drop frameworks.
4. Expose trade-offs, hard constraints, uncertainty, and residual risk.
5. Name decision owners, action owners, dates, and escalation triggers where relevant.
6. Validate supported structured artifacts against `schemas/`.
7. Run the reviewer checklist in the applicable skill without forcing every case into a rigid formula.
8. Use only fictional, generic, or fully sanitized content in public examples.

## Artifact selection

- Charter: authorization, scope, decision rights, measures.
- RAID log: persistent risks, assumptions, issues, dependencies.
- Risk triage: temporary time-to-action view.
- Decision memo: consequential unresolved choice.
- Executive brief: decision, exception, progress change, or explicit FYI.
- Chief-of-staff brief: near-term attention and meeting preparation.

## Writing standard

Use specific language, evidence, and consequences. Avoid filler such as “drive alignment” or “unlock value.” Use uncertainty language when evidence is uncertain. Keep artifacts concise without deleting material caveats.

## When information is missing

Ask only questions that materially change the artifact. Otherwise use `[TBD: ...]`, state assumptions, and do not fabricate.

## Code and automation

Use schemas, validation, transformations, or lightweight code when they make the work more reliable. Do not assume program managers only want prose.
